# References

## 역할
`references/`는 프로젝트를 가로지르는 참고 문서 원문을 보관하는 레이어다.
이 디렉터리는 작업을 직접 설명하는 곳이 아니라, 작업이 따라야 하는 배경 문서를 찾는 곳이다.

포함 대상:
- 기획 문서
- 구조 원칙 문서
- 기능 명세 문서
- 테스트 기준 문서
- 정책 문서
- 외부 참고 문서

## 언제 읽는가
대부분의 작업시 참고, 특히 아래 조건이면 거의 필수.

반드시 읽는다:
- 사용자가 특정 문서, 기술 패턴, 정책, PRD를 직접 언급한 경우
- 작업 의도가 `plan`, `add`, `refactor`, `govern`인 경우
- 작업 범위가 `cross-cutting`, `global`인 경우

강하게 권장한다:
- `investigate`
- `fix`
- `change`

이유:
- feature 문서는 현재 구조를 알려주지만, references는 왜 그 구조를 따라야 하는지와 어떤 제약이 있는지를 알려준다.

## 접근 순서
1. `index.yaml`이 있으면 먼저 읽는다.
2. `kind`, `tags`, `path`로 관련 문서를 좁힌다.
3. 좁혀진 문서의 상대경로를 따라 원문 파일을 연다.
4. 원문은 필요한 부분만 읽는다.

## 디렉터리 구조
문서는 종류별 하위 디렉터리에 둔다.

- `architecture/`
- `product/`
- `specification/`
- `testing/`
- `policy/`
- `external/`

## 문서 종류
`index.yaml`의 `kind`는 아래처럼 쓰고, 각 값은 하위 디렉터리 이름과 1:1로 맞춘다.

- `product`
  - 요구사항, 사용자 시나리오, 제품 목표
- `architecture`
  - 구조 원칙, 레이어 규칙, 공통 패턴
- `specification`
  - 특정 기능이나 화면, 도구에 대한 상세 명세
- `testing`
  - 테스트 시나리오, 검증 기준, 품질 체크리스트
- `policy`
  - 저장소 운영 규칙, 문서 관리 규칙, 승인/예외 처리 규칙
  - 제품 기능 명세가 아니라 작업 절차와 거버넌스를 다룬다
- `external`
  - 외부 공식 문서에 대한 발췌, 정리, 벤더/프레임워크 참고 메모
  - 프로젝트 내부 규칙이 아니라 외부 시스템이나 기술에 대한 참고다

## index 작성 규칙
- 경로는 `references/` 기준 상대경로로 적는다.
- 경로는 반드시 종류 디렉터리를 포함한다.
- 문서 하나당 record 하나를 둔다.
- `id`는 전역에서 겹치지 않게 쓴다.
- `title`은 사람이 바로 이해할 수 있게 쓴다.
- `kind`는 위 분류 중 하나만 쓴다.
- `tags`는 검색과 라우팅용으로만 쓴다.
- 특정 feature와의 연결은 이 파일에서 직접 소유하지 않는다.
- feature 연결은 `features/<feature-id>/meta.yaml`의 `related_references`와 해당 feature의 `current.md`가 맡는다.

## 태그 작성 규칙
태그는 전부 소문자 `namespace:value` 형태로 쓴다. 항상 넣는 태그들은 다음과 같다.

- `kind:*`
- `intent:*`
- `scope:*`
- `topic:*`

외에도 필요한 경우 추가 태그 작성이 자유롭게 가능하다.


## 운영 규칙
- 새 원문 문서를 추가하면 `index.yaml`도 같은 커밋에서 갱신한다.
- 문서의 실제 성격과 `kind`가 다르면 디렉터리부터 다시 배치한다.

## 문서가 아직 없는 경우
- 실제 문서가 아직 없으면 해당 종류 디렉터리에 새 파일을 만들고 `index.yaml`에 추가한다.
- 기준이 없으면 `index.yaml.template`을 먼저 참고한다.
